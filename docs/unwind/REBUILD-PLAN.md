## Target Stack

**From:** Plain JavaScript (CommonJS), Node `>=14`, no build step; built on `amqplib`; tested with zUnit + nyc.
**To (assembled in the interview — accept-all-recommended):**

| Element | Choice | From (current) |
|---------|--------|----------------|

| Packaging / module format | **Dual ESM + CJS** with `exports` map (tsup) | CommonJS only, `main: index.js`, no build |
| Public API contract | **Keep both** callback `Broker` + promise `BrokerAsPromised`; async/await core | Dual API over callback core (`async.compose`) |
| Underlying transport | **Keep `amqplib`** (peerDependency) | `amqplib` peer dep |
| Datastore / data access | **N/A** — no persistence | none (ephemeral in-memory state) |
| API contract style | **N/A** — no HTTP surface | library exports + RabbitMQ Management HTTP client |
| Frontend | **N/A** — backend library | none |
| Dependency modernization | Replace `async`, `uuid`, `safe-json-parse`, `forward-emitter`, `xregexp`, trim `lodash`; keep `generic-pool`, `lru-cache`, `debug` | 10 runtime helper deps |
| Testing | **Vitest** (+ v8 coverage), retain docker RabbitMQ harness | zUnit + nyc |
| Hosting / distribution | **npm publish** (release-triggered GitHub Actions) | same |
| Architecture | Single package, **preserve layer boundaries** | layered `lib/` modules |

**Changing:** language (→ TS), build/packaging (→ dual ESM+CJS), internal control flow (callbacks → async/await), helper deps, test runner.
**Staying:** the public export contract (both APIs), `amqplib` as transport, the topology-assertion task pipeline, the config schema, `generic-pool` channel pooling, the real-broker test strategy.
**Architecture:** modular monolith, one published package; the discovered layers map 1:1 onto TS modules:
`config/` (domain + validation) → `backoff/` + `counters/` (service) → `amqp/` + `amqp/tasks/` (messaging) → `index.ts` (public surface) + `management/` client.

## Executive Summary

**Overall Readiness:** 100% — all six analyzed layers verified to 100% coverage; **no provisional layers**.
**Recommended Approach:** A faithful, **non-breaking** TypeScript re-implementation that preserves the dual public API and `amqplib` semantics, modernizes the internals (async/await, native equivalents for helper libs), and ships dual ESM+CJS — validated by porting the existing real-broker integration suite and parallel-running old vs new.

This is a **library rebuild**, not an application migration: there is no database, HTTP API, frontend, or hosting target to re-platform (brief confirms 0 data models, 0 endpoints, no ORM). The strategic surface is therefore narrow and well-pinned — the risk is behavioral fidelity to RabbitMQ, not architecture.

## Strategic Decisions

> Each decision was made during the rebuild interview (accept-all-recommended mode) and is recorded in `## Rebuild Decisions` and `.cache/rebuild-decisions.json`. Assessments cite the deterministic brief.

### Language, Packaging & API Contract (RD-1, RD-2, RD-3)
**Decision:** TypeScript full rewrite · dual ESM+CJS · keep both public API surfaces.
**Why:** The library is untyped CJS with a large `require()` install base. TS adds shipped `.d.ts` and typed config; dual output keeps every existing consumer working while enabling `import`; keeping both `Broker` (callbacks) and `BrokerAsPromised` (promises) makes the rewrite non-breaking. The 9 `index.js` exports are the `[MUST]` contract — see [layers/api/index.md](layers/api/index.md).

### Underlying Transport (RD-4)
**Decision:** Keep `amqplib` as a peer dependency.  **Why:** rascal exists to wrap amqplib; its channel/connection/confirm semantics permeate every layer. Swapping it would be a different library, not a rebuild.

### Dependency Modernization (RD-5)
**Decision:** Replace helpers with native equivalents where safe; keep specialized infra libs.
**Why:** Modern Node/TS removes the need for several deps. Replace: `async`→native async/await, `uuid@8`→`crypto.randomUUID`, `safe-json-parse`→`try/catch`, `forward-emitter`→explicit re-emit, `xregexp`→native named-group regex, and trim `lodash` (use `structuredClone` for cloneDeep). Keep: `generic-pool` (channel pools — see [layers/messaging/vhost.md](layers/messaging/vhost.md)), `lru-cache` (counters — see [layers/service-layer/index.md](layers/service-layer/index.md)), `debug`. Note the flagged `inMemoryCluster.js` `key`-mismatch finding — fix during the port, don't reproduce it.

### Testing & Validation (RD-6, RD-12, RD-14)
**Decision:** Port the 15 suites to Vitest, preserving assertions; run old + new against one real RabbitMQ.
**Why:** The `tests` layer is at 100% coverage and is integration-style (real broker) — the strongest equivalence asset available. Vitest replaces zUnit+nyc with TS-native tests and built-in coverage. See [layers/unit-tests/index.md](layers/unit-tests/index.md).

### Not Applicable
**Database strategy / Frontend retention / Integration preservation / Hosting:** N/A — no DB, UI, external integrations, or hosting target. Distribution is via the existing npm publish workflow (update the Node version matrix).

## Phasing Strategy

> Order seeded by `importGraph.foundations` (most depended-upon leaves first) and `[MUST]` weighting. Every phase references the verified layer docs for the exact contracts.

### Phase 1: Foundations — config & primitives
**Build first:** `lib/config/*` (baseline, defaults, fqn, configure, validate, schema), `lib/utils/setTimeoutUnref`, `lib/counters/*`, `lib/backoff/*`.
**Approach:** Rebuild as typed modules; turn `schema.json` into TS config interfaces + a runtime validator. These are the import-graph leaves (`config/tests` has 9 dependents, `setTimeoutUnref` 5, `counters/inMemory` 4).
**Reference:** [layers/domain-model/index.md](layers/domain-model/index.md), [layers/service-layer/index.md](layers/service-layer/index.md).
**Validation:** port `config.tests`, `defaults.tests`, `validation.tests`, `backoff/*.tests`, `caches/inMemory.tests` → green.

### Phase 2: Core messaging — transport & topology
**Build:** `lib/management/Client` (3 dependents), then `lib/amqp/tasks/*` (connection/channel/topology assert/check/delete/init), then `Vhost` (channel pools, reconnection/backoff), then `Broker` (the orchestration hub, 2 dependents).
**Approach:** Re-express the `async.compose` task chains as ordered async/await pipelines, preserving the documented chain order exactly. Highest-`[MUST]` density (messaging: 36 candidates).
**Reference:** [layers/messaging/index.md](layers/messaging/index.md) (broker / vhost / tasks / recovery).
**Validation:** `vhost.tests`, `broker.tests`, `shovel.tests` against real RabbitMQ.

### Phase 3: Pub/Sub & recovery
**Build:** `Publication` / `PublicationSession`, `Subscription` / `SubscriberSession`, `SubscriberError` (recovery strategies: ack/nack/republish/forward/dlq + defer), `XDeath`.
**Reference:** [layers/messaging/publication.md](layers/messaging/publication.md), [layers/messaging/subscription.md](layers/messaging/subscription.md), [layers/messaging/recovery.md](layers/messaging/recovery.md).
**Validation:** `publications.tests`, `subscriptions.tests` (+ recovery/redelivery assertions).

### Phase 4: Public surface & promise wrappers
**Build:** `BrokerAsPromised`, `SubscriberSessionAsPromised`, then `index.ts` re-exporting the full 9-key surface; wire the `exports` map for dual ESM+CJS.
**Reference:** [layers/api/index.md](layers/api/index.md).
**Validation:** `brokerAsPromised.tests`, `publicationsAsPromised.tests`, `subscriptionsAsPromised.tests`; dual-format smoke test (`require('rascal')` and `import 'rascal'` both resolve all 9 exports).

### Phase 5: Packaging, CI & cutover
**Build:** tsup dual build, `.d.ts` emission, update `package.json` (`exports`, `types`, `engines`), migrate CI to Vitest, update the Node matrix, keep the release→npm-publish workflow.
**Reference:** [layers/infrastructure/index.md](layers/infrastructure/index.md).
**Cutover:** parallel-run legacy vs rebuilt against representative configs; ship as a major version with a migration note (TS types added, deps changed, behavior preserved).

## Validation Strategy

**Equivalence testing:** the ported integration suite IS the equivalence harness — it asserts real RabbitMQ behavior, so a green run proves parity. Coverage baseline: 100% of the `tests` layer (15 suites).
**Test vectors:**
- [ ] All 15 ported suites pass on Node LTS matrix (real broker)
- [ ] Publish/subscribe/recovery behavior identical (acks, nacks, redelivery counts, dead-letter routing)
- [ ] Vhost topology assertion + reconnection/backoff behavior identical
- [ ] Dual-format resolution: `require` and `import` both expose all 9 exports with matching shapes
**Go-live checklist:**
- [ ] Each phase's validation green
- [ ] Parallel old-vs-new run on representative configs
- [ ] `inMemoryCluster` key-mismatch finding fixed (not reproduced)
- [ ] `engines.node` / CHANGELOG consistency resolved
- [ ] Rollback = unpublish/deprecate the major; consumers pin previous
